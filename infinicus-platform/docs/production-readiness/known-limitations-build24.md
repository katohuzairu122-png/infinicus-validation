# BUILD-24 — Secrets and Configuration Management: Known Limitations

1. **No real managed secret store is integrated.** `EnvSecretProvider` (reads `process.env`) is the only `SecretProvider` implementation — no cloud secret manager (AWS Secrets Manager, HashiCorp Vault, Doppler, etc.) is reachable from this sandboxed development environment, the same class of constraint documented for BUILD-22's cloud backup targets and BUILD-23's container registry. `SecretProvider` is the seam: a production deployment implements the same two-method interface against a real service without changing `loadConfig()`, `redactSecretValues()`, or any other call site.

2. **No automated/scheduled rotation.** `rotate-db-credential.sh` and `secret-rotation-audit.cjs check-expiration` are manually-invoked, live-verified tools — there is no cron job or CI schedule triggering rotation automatically. This mirrors BUILD-22's PITR drill: proven to work for real, but operationally triggered by a human (or a future ops-automation build), not on a timer.

3. **`check-no-browser-secrets.mjs` is a static source scan, not a built-bundle scan.** It inspects `apps/web`/`apps/admin`'s TypeScript/JSX source directly. It does not additionally parse the compiled `.next` client bundle to verify no secret substring survived Next.js's build pipeline by some indirect path this scanner's pattern-matching didn't anticipate (e.g. a secret re-exported through several layers of indirection before reaching a `NEXT_PUBLIC_` reference). Source-level scanning catches every case this platform's actual code exhibits today; bundle-level scanning would be a stronger but more expensive defense-in-depth layer, out of scope here.

4. **`SECRET_INVENTORY` is hand-maintained, not derived automatically from source.** A new environment variable introduced in server-side code must be added to `SECRET_INVENTORY` by whoever adds it — there is no static-analysis step that fails CI if a `process.env.X` reference exists with no matching inventory entry. The existing `.env.example` staleness this build fixed is exactly the failure mode this limitation describes; nothing in this build prevents it from recurring for a *newly introduced* variable (as opposed to the pre-existing drift this build corrected).

5. **The production-credential guard is a heuristic, not a certainty.** `looksLikeLocalOrTestCredential()` matches known local/CI patterns (hostnames, disposable test usernames/passwords) — a production `DATABASE_URL` that happened to coincidentally contain one of these substrings (extremely unlikely given the specificity of the patterns, e.g. `local_admin_pw`) would be incorrectly refused; conversely, a genuinely wrong non-local credential that doesn't match any known pattern would not be caught by this guard. It is a real, tested safety net for the specific failure mode it targets (accidentally running production against the shared local/CI database), not a general credential-correctness validator.

6. **Rotation applies to the database credential only.** This build's rotation/expiration tooling (`rotate-db-credential.sh`, `secret-rotation-audit.cjs`) is generic at the audit-table/CLI level (`secretName` is a free-form parameter), but only `DATABASE_URL`'s actual rotation mechanism (`ALTER ROLE ... VALID UNTIL`) is implemented. There are currently no other secrets in this platform's real inventory requiring rotation (see configuration-build24.md) — `ADMIN_DATABASE_URL` shares the same underlying database-role rotation mechanism when needed.
