# BUILD-26 ŌĆö Security, Privacy, and Retention: Known Limitations

1. **No full commercial DAST suite.** `dast-scan.sh` is a genuine but narrow black-box scan (security headers, stack-trace leakage, SQLi/XSS rejection, unauthenticated-admin-route rejection) ŌĆö it is not OWASP ZAP, Burp Suite, or an equivalent full crawler-based scanner, none of which are installable from a reachable source in this sandboxed environment. It covers what is real, automatable with `curl`, and meaningful for this specific API's shape (JSON-only, Bearer-token auth, no HTML views) ŌĆö not an exhaustive security assessment.

2. **`vitest`'s critical CVE (GHSA-5xrq-8626-4rwp) is not patched, only assessed as unreachable.** The fix requires bumping `vitest` from `^1.6.0` to `>=3.2.6` ŌĆö a major-version upgrade across every package.json in this 24-package monorepo, with a real risk of breaking the ~2800 already-passing tests written against vitest 1.x's API surface, for a vulnerability that requires `vitest --ui` (a flag this codebase's `"test"` scripts never pass, verified across every package.json). Documented and allowlisted, not silently ignored ŌĆö revisiting this decision is recommended whenever a future build touches the test infrastructure directly (a natural candidate: a future testing-infrastructure-focused build, not invented here).

3. **`sharp`'s high-severity advisory is assessed as unreachable, not patched.** Transitive via Next.js's `next/image` component, which `apps/web` never imports. If a future build adds image handling to `apps/web`, this assessment must be re-verified ŌĆö the advisory is real and would become genuinely exploitable the moment `next/image` (or `sharp` directly) is actually used.

4. **`delete-tenant-data.mjs` discovers tenant-scoped tables dynamically but does not verify RLS policy *safety*, only *presence*.** The script correctly distinguishes tables with a literal `tenant_id` column (safe for an explicit `WHERE tenant_id = ...` filter) from those scoped indirectly (currently just `identity.api_key_references`, verified safe by inspection) ŌĆö but if a *future* migration adds a new tenant-scoped table with an unusual RLS pattern this script's two-category logic doesn't anticipate, that table would need the same manual verification this build applied to `identity.api_key_references`. This is a real, bounded risk inherent to any schema-introspection-driven script; the mitigation is the automated test (`delete-tenant-data.integration.test.ts`) asserting the safety property (a second tenant + shared system roles survive) on every CI run, which would catch a regression the moment it's introduced, not just at authoring time.

5. **No automated retention-window policy.** This build delivers the deletion *mechanism* (`delete-tenant-data.mjs`), live-verified and safe ŌĆö there is no scheduler or policy engine deciding *when* a tenant should be automatically erased (e.g., "N years after account closure"). Every invocation today is a deliberate, manually-triggered operator action in response to an explicit request.

6. **CSRF protection is genuinely not applicable**, not merely unimplemented ŌĆö documented at length in security-controls-build26.md. If a future build ever introduces cookie-based session authentication (a real architectural change from today's Bearer-token-only design), this assessment must be revisited and CSRF tokens added at that time.

7. **The threat model (architecture-and-scope-build26.md) is a living document scoped to this build's own audit**, not a formally maintained, continuously-updated artifact with its own review cadence ŌĆö a future build or process should own keeping it current as the platform's attack surface grows.
