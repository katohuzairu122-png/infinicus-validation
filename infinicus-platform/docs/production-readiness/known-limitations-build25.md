# BUILD-25 — Logging, Monitoring, and Alerting: Known Limitations

1. **No real APM/error-tracking backend integrated.** `LoggingErrorTracker` is the only `ErrorTracker` implementation — no Sentry, Bugsnag, or similar service is reachable from this sandboxed environment. `ErrorTracker` is the seam: a production deployment implements the same one-method interface against a real backend's SDK (or composes it via `CompositeErrorTracker` alongside `LoggingErrorTracker`) without changing any call site.

2. **No real distributed-tracing backend integrated.** `startSpan()` records span start/end/duration as structured log lines — genuinely functional and reconstructable by grepping logs for a shared `traceId`, but there is no Jaeger/Tempo/OTel-collector export, and no trace-visualization UI. This is the seam an OpenTelemetry SDK integration attaches to later.

3. **No real dashboard UI.** `GET /v1/metrics` is a real, live-tested JSON data source — but no Grafana (or similar) instance is stood up to visualize it. A future build or external tool consumes this endpoint directly.

4. **No true SLO/availability percentage.** `observability-audit.cjs summary` reports real, computed counts (errors, outbox backlog, active alerts) but deliberately does not compute an availability or error-rate *percentage* — that requires a request-volume denominator (total requests served in the window) that no table in this platform currently tracks. Fabricating a percentage without that denominator would violate this platform's "execution evidence, not fabricated status" principle (root `CLAUDE.md` §10). A future build that adds request-volume metrics (candidate: BUILD-27 Performance) would be the natural place to add a genuine SLO percentage on top of this build's error/outbox counting.

5. **No automated/scheduled alerting.** `observability-audit.cjs`'s threshold-check commands are manually-invoked, live-verified tools with clean exit-code semantics — there is no cron job, CI schedule, or paging-service integration (PagerDuty, Opsgenie) triggering them automatically or routing their output anywhere. This mirrors BUILD-22's PITR drill and BUILD-24's rotation script: proven to work for real, operationally triggered by a human or a future automation build, not on a timer.

6. **`GET /v1/metrics`'s error/outbox figures are RLS-scoped, not a true cross-tenant platform aggregate**, when called through the application's own least-privilege database role (which every real HTTP request necessarily does). The endpoint deliberately resets to a nil-tenant session context first, so it reports tenant-NULL-scoped counts only — a real, deterministic, but partial view. A true cross-tenant aggregate requires `observability-audit.cjs` run with `ADMIN_DATABASE_URL` (a CLI/ops path, not an HTTP one) — documented, not hidden.

7. **`error_events` does not store stack traces or full request payloads.** Only `errorName`/`message`/`route`/`statusCode`/`correlationId`/`context` are persisted — deliberately lean, to minimize the surface area for accidental secret/PII leakage into a queryable table. Full diagnostic detail (stack trace, request body) remains in the structured logs only, correlated via `correlationId`.
