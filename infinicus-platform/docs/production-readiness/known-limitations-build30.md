# BUILD-30 — Production Acceptance and Launch: Known Limitations

- **This checklist evidences a sandboxed development environment, not a real production deployment.** Every live check in this build (load test, smoke test, DAST scan, restore drill) ran against the same local PostgreSQL instance and in-process Fastify server this entire session has used — not a real staging/production infrastructure with its own network topology, load balancer, managed database, or DNS. The methodology transfers; the specific numbers (load-test latencies especially) should be re-measured against real target infrastructure before being trusted as production capacity commitments — the same caveat BUILD-27's own known-limitations doc already states.

- **No real production promotion was executed.** This build produces the acceptance record and a recommendation to approve staging; it does not itself run `deploy.sh` against a real production `DATABASE_URL`/hosting target, since no such target exists in this sandboxed environment. "Staging approval" in the launch checklist is a documented human-decision gate, not evidence that a real promotion has occurred.

- **One load-test run's p99 marginally exceeded its SLO target** (113.7ms vs. a 100ms target), before two subsequent consecutive runs cleared it comfortably (88.5ms, 89.7ms). Documented honestly in the launch checklist and test-evidence-build30.md rather than omitted or explained away — attributed to sandboxed-environment resource variance, not a systematic defect, but a real reviewer should weigh this when deciding how much margin the ≤100ms target actually has in this environment.

- **No full commercial DAST/penetration-test pass, no chaos-engineering drill.** Both are known limitations already carried forward from BUILD-26 (no OWASP ZAP/commercial DAST tooling installable in this sandboxed environment) and BUILD-29 (no genuine "kill the database mid-request" chaos drill performed) respectively. This build does not close either gap — it only carries them forward honestly into the final acceptance record rather than letting a "launch checklist" imply more assurance than actually exists.

- **The dependency-scan allowlist now covers 6 advisories total** (2 from BUILD-26, 4 new in this build). This is expected to grow over time as dependency versions shift and new advisories are published — the allowlist is a living document requiring periodic review (e.g. before each future production promotion, not just at initial launch), not a one-time artifact. A real operating process should schedule a recurring dependency-scan re-run (the lifecycle-audit script precedent from BUILD-28's `billing-lifecycle-audit.cjs` — external scheduling required, no in-process job runner) rather than relying solely on it being caught incidentally at a future build's own acceptance check.

- **`launch-acceptance-check.mjs` does not itself run the security-gate, restore-proof, privacy-proof, or critical-workflow checks** — those are run as separate commands (see operating-procedure-build30.md), not folded into the one script's single invocation. This was a deliberate scope decision (each of those already has its own well-tested, independent tool; folding them into one giant script would either duplicate logic or create a single point of failure across unrelated concerns) but means a release manager must run more than one command to reproduce the full acceptance matrix, not just one.
