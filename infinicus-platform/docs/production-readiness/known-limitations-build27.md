# BUILD-27 — Performance and Load Readiness: Known Limitations

- **Outbox emission is not wired into any domain write path.** Confirmed by exhaustive grep across every layer/package: `simulation.emit_scenario_created`, `emit_run_requested`, `emit_run_started`, and their equivalents in every other domain (`ai_decision_intelligence`, `business_digital_twin`, etc.) are real, tested SQL primitives with zero call sites outside their own migration files. Domain writes (`createModel`, `createScenario`, `createRequest`, `createRun`, ...) commit successfully but emit no outbox event today. This build's outbox-throughput test measures the real emission primitive directly (`simulation.emit_run_requested()` via SQL) since that is the only genuinely exercised code path — it does not, and cannot yet, measure "outbox events emitted per domain write" because that capability does not exist. **Recommendation**: a future build should wire `emit_*` calls into each domain repository's mutating methods (likely inside the same transaction as the write itself, matching the pattern each `emit_*` function already assumes) — this is a cross-cutting change touching every layer, not a performance-testing concern, and is out of BUILD-27's scope.

- **Right-to-erasure cannot fully erase a tenant with real audit-trail history.** ~210 tables across every domain (intake status-history, run status-history, versions, evidence, deployment-rollback tables, ...) are deliberately append-only (`forbid_mutation` trigger — no role can `UPDATE`/`DELETE`), by design, to protect audit-trail integrity. `delete-tenant-data.mjs` now handles this gracefully (skips these tables, reports them as retained, still erases everything that legitimately can be erased, records the full breakdown in `platform.data_deletion_events`) rather than crashing — but a tenant with real usage in these domains genuinely **cannot** be made to disappear entirely via `DELETE`. This is a real, unresolved tension between the platform's audit-integrity guarantee and a literal reading of GDPR Article 17 "right to erasure" for such a tenant. **Recommendation**: a future build should design an explicit anonymization strategy for append-only audit-trail rows (replace personally-identifying fields with a redacted placeholder while preserving row/event shape for audit continuity) rather than physical deletion, likely requiring legal/compliance input on which fields in which tables actually carry personal data versus operational metadata. This is a data-governance design decision, not a mechanical fix, and is explicitly out of scope for a performance build.

- **This build's measured numbers reflect one sandboxed, shared-tenancy development environment**, not a dedicated production host. Throughput/latency numbers should be re-measured against the actual target production infrastructure before being used for firm capacity commitments; the methodology and tooling (`load-test.mjs`, the live concurrency test suites) are the durable deliverable, not the specific millisecond figures.

- **No distributed/multi-node load testing.** The platform runs as a single `apps/api` process against a single Postgres instance in every environment available in this build; horizontal-scaling behavior (multiple API instances behind a load balancer, read replicas, connection pooler like PgBouncer) is untested and out of scope here.

- **No load testing of authenticated, write-heavy, multi-hop workflows** (e.g. a full BI→DT→Simulation→ADI→ABA handoff chain under concurrent load) — this build measures each domain's request/run-creation step in isolation (consistent with the "shallow chain" pattern already established for Simulation/ADI fixture setup in prior builds) and the platform's raw HTTP/DB primitives, not an end-to-end cross-layer workflow under load. A future build focused on end-to-end workflow performance would need its own fixture design.

- **`load-test.mjs` uses Node's built-in `fetch`, not a dedicated load-testing tool** (autocannon, k6, wrk). This was a deliberate choice (no new dependency for a tool whose results are read once and recorded, and this sandboxed single-process environment cannot usefully drive load beyond what Node's own concurrency model achieves) — for genuinely high-throughput production load testing, a dedicated external tool would give more accurate results at concurrency levels beyond what a single Node process can realistically generate.
