# BUILD-29 — Incident Response and Rollback: Known Limitations

- **No external paging/on-call scheduling integration.** `docs/incident-response/on-call-ownership.md` defines the model and escalation path, but this build does not integrate with PagerDuty, Opsgenie, or any external paging service — there is no mechanism in this codebase that actually notifies a human when an incident is declared. A real deployment needs to wire `POST /v1/incidents` (or a webhook off `platform.incidents`) to a real paging service.

- **No public-facing status page.** `communication-templates.md` defines what a customer-facing (`is_customer_facing: true`) timeline entry should say, and the data model supports querying just those entries, but this build does not publish them anywhere external — no status-page application or integration exists. A real deployment would build a thin read-only feed off `GET /v1/incidents`/`GET /v1/incidents/:id/updates` (filtered to `is_customer_facing: true`) for that purpose.

- **Coarse permission gate.** Every incident route uses the single `platform:admin` permission — there is no `incident:read` vs `incident:write` split, and no way to grant a support engineer read-only incident visibility without granting them every other `platform:admin` capability too (billing management, metrics, etc.). Proportional to this build's scope; a larger operator team would want a finer split.

- **No automatic incident detection.** Incidents are declared by a human via the API. `observability.alert_events` (BUILD-25) already exists and could plausibly feed an auto-declare capability (e.g. N consecutive critical alerts auto-create a Sev2 incident), but that wiring is not built here — this build's incident tracking is a manual-declaration system, not an alerting system.

- **Severity is not retroactively editable.** If an incident's true severity turns out to be different from its initial declaration, the documented process (severity-model.md) is to note the reassessment in a timeline update — there is no `PATCH`-style "change the severity" operation. This was a deliberate design choice (the severity column reflects the declaration-time assessment; the append-only timeline shows how understanding evolved) but means a dashboard sorting purely by the `severity` column will show the original assessment, not the current one, unless it also reads the timeline.

- **Runbooks are documentation, not automation.** `deployment-rollback.md`/`migration-rollback.md`/`restore-procedure.md`/`security-incident.md`/`provider-outage.md` are all step-by-step human-followed procedures referencing existing scripts — none of them is itself a single "run this one command to fully remediate" tool. This is intentional (an incident's actual remediation requires judgment a runbook can guide but not replace) but is worth stating plainly: this build does not add one-click incident remediation.

- **No incident-response-specific load/chaos testing.** This build's own tests verify the tracking mechanism (declare/update/resolve/timeline) works correctly under normal conditions; it does not include a genuine chaos-engineering drill (e.g. actually killing the database mid-request to verify the readiness-based provider-outage detection works as documented) — BUILD-22's own WAL-archiving/PITR live drill (test-evidence-build22.md) is the closest prior-build precedent for that kind of test, and this build's restore-procedure runbook explicitly defers to that evidence rather than re-deriving it.
